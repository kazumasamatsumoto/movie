# #160 「`implements`って積極的に書いてる？」制作ドキュメント

## 台本
ずんだもん「`implements`って積極的に書いてる？」
四国めたん「v5.9では`implements`は型の契約を明示し、クラスが特定のインターフェースを満たすことを保証します」
ずんだもん「ぼくは公開APIやプラグインシステムでは必ず`implements`を書いて、契約を明確にするんだよね」
四国めたん「`implements`を使うと、インターフェースの変更時にクラスの実装不足が検出され、型安全性が高まります」
ずんだもん「プラグインインターフェースを`implements`で実装したら、必要なメソッドの実装漏れがコンパイル時に見つかったんだよ！」
四国めたん「みなさんは`implements`を積極的に使っていますか？」
ずんだもん「使う場面や効果をコメントで教えてね！」

## 画面表示用コード

```typescript
// インターフェースで契約を定義
interface Plugin {
  name: string;
  version: string;
  initialize(): void;
  execute(context: any): Promise<void>;
  cleanup(): void;
}

// implementsで契約を保証
class MyPlugin implements Plugin {
  name = 'MyPlugin';
  version = '1.0.0';

  initialize(): void {
    console.log('Initializing plugin');
  }

  async execute(context: any): Promise<void> {
    // 実装
  }

  cleanup(): void {
    console.log('Cleaning up');
  }

  // メソッドが足りないとコンパイルエラーになる
}
```

```typescript
// 複数のインターフェースを実装
interface Loggable {
  log(message: string): void;
}

interface Configurable {
  configure(config: any): void;
}

class Service implements Loggable, Configurable {
  log(message: string): void {
    console.log(message);
  }

  configure(config: any): void {
    // 設定処理
  }
}
```

```typescript
// implementsなしでも動くが、契約が明示されない
class UnsafePlugin {
  // Pluginインターフェースを満たしているか不明確
  name = 'Unsafe';

  initialize() {
    // ...
  }
  // executeやcleanupがなくてもエラーにならない
}
```

## 制作メモ
- **問いの切り口**: implementsの積極的使用
- **想定視聴者層**: 契約ベースの設計を行うエンジニア
- **押さえる技術キーワード**: implements, interface, 契約, 型安全性
- **キャラの立場**: 四国めたん=契約保証の役割解説 / ずんだもん=実装漏れ検出の効果
- **コメントで聞きたいこと**: 使用場面や効果
- **補足メモ**: プラグインシステムという実用的な文脈で説明
