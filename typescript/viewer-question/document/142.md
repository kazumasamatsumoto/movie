# #142 「`TData`とか`TError`の命名、どう思う？」制作ドキュメント

## 台本
ずんだもん「`TData`とか`TError`の命名、どう思う？」
四国めたん「v5.9では`T`プレフィックスを付ける命名は賛否両論で、プロジェクトによって方針が分かれます」
ずんだもん「ぼくは`T`プレフィックスがあると型パラメータだと一目でわかるから好きなんだよね」
四国めたん「`T`プレフィックスは型パラメータであることを明示しますが、冗長に感じる人もいます」
ずんだもん「`Data`という具体的な型と`TData`という型パラメータが混在するコードで、`T`プレフィックスのおかげで混乱しなかったんだよ！」
四国めたん「みなさんは`T`プレフィックスをどう思いますか？」
ずんだもん「命名の好みや理由をコメントで教えてね！」

## 画面表示用コード

```typescript
// Tプレフィックス派
interface ApiClient<TRequest, TResponse, TError = Error> {
  call(request: TRequest): Promise<TResponse>;
  handleError(error: TError): void;
}

// 型パラメータだと一目でわかる
type Result<TData, TError> =
  | { success: true; data: TData }
  | { success: false; error: TError };
```

```typescript
// プレフィックスなし派
interface ApiClient<Request, Response, Err = Error> {
  call(request: Request): Promise<Response>;
  handleError(error: Err): void;
}

// よりシンプルで読みやすい
type Result<Data, Err> =
  | { success: true; data: Data }
  | { success: false; error: Err };
```

```typescript
// 混在する場合の利点
type Data = { id: string; name: string }; // 具体的な型

interface Container<TData> {
  value: TData;
  defaultData: Data; // 具体的な型と区別できる
}
```
