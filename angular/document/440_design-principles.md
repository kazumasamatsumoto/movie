# #440 「カスタム Directive の設計原則」

## 概要
カスタムディレクティブの設計原則は責務の明確化、拡張性、テスト容易性、プラットフォーム非依存、ドキュメント整備を柱とし品質を保証する。

## 学習目標
- 設計原則の全体像を理解する
- 各原則を実装に落とし込む方法を学ぶ
- チーム標準として原則を共有する際のポイントを把握する

## 技術ポイント
- 単一責務(SRP)とInput/Outputの明確化
- Renderer2/PLATFORM_IDで環境依存を排除
- テスト・ドキュメント・リリースポリシーを整備

## 📺 画面表示用コード（動画用）
```markdown
- 責務分離
- 拡張可能なAPI
- テスト容易性
- プラットフォーム非依存
- ドキュメント整備
```

## 💻 詳細実装例（学習用）
```markdown
1. 責務の明確化: 見た目変更のみ → Directive、データ取得 → サービス
2. 拡張性: Inputで設定、Outputでイベントを通知
3. テスト容易性: ホストコンポーネントでユニットテスト
4. プラットフォーム非依存: Renderer2とisPlatformBrowserでガード
5. ドキュメント: README/Storybookで使用方法共有
```

## ベストプラクティス
- 原則をドキュメント化し、コードレビューでチェック項目にする
- 設計に迷った場合は原則に立ち返り、責務の過不足を確認
- 原則はプロジェクトの成熟度に合わせてアップデートする

## 注意点
- 原則を守るために過度な抽象化を行わない
- チーム内で共通認識がなければ意味が薄れるため教育を行う
- 原則違反のケースは例外として明記し、理由を共有
